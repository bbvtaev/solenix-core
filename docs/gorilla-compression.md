# Реализация Gorilla-сжатия

> Пакет: `internal/chunk`
> Файлы: `gorilla.go`, изменения в `writer.go` и `reader.go`
> Источник алгоритма: Facebook "Gorilla: A Fast, Scalable, In-Memory Time Series Database" (2015)

---

## Что это и зачем

Chunk-файлы хранят исторические временные ряды. До версии `0x02` каждая точка занимала
ровно 16 байт: `int64` (timestamp) + `float64` (value). Gorilla-сжатие использует
статистические свойства временных рядов:

- **Временные метки** обычно идут с постоянным шагом → Delta-of-Delta ≈ 0 → 1 бит вместо 8 байт.
- **Значения** меняются плавно → XOR соседних `float64` имеет мало значащих бит.

Реальные коэффициенты сжатия (100 точек, 15s интервал):

| Тип данных | Raw | Gorilla | Сжатие |
|---|---|---|---|
| Константное значение | 1600 B | 49 B | **32.7x** |
| Медленно растущий счётчик | 1600 B | 142 B | **11.3x** |
| Линейно меняющиеся float | 1600 B | 830 B | **1.9x** |
| Случайный интервал + случайные float | 1600 B | 1574 B | **~1x** |

## Алгоритм

### Побитовая запись/чтение

Всё сжатие работает на уровне отдельных битов, поэтому базовые примитивы —
`bitWriter` и `bitReader`. Биты пакуются **MSB-first** (старший бит первым) в каждый байт.

```
bitWriter:
  cur   byte   — текущий накапливаемый байт
  nBits uint8  — сколько бит уже заполнено в cur (0..8)

  writeBit(b):
    если nBits == 8: сбросить cur в buf, обнулить
    установить бит на позиции (7 - nBits) в cur
    nBits++

  writeBits(v uint64, n uint8):
    записать n бит числа v, начиная с MSB

  bytes():
    вернуть buf + cur (если nBits > 0)
```

```
bitReader:
  buf   []byte
  pos   int    — индекс следующего байта для загрузки
  cur   byte   — текущий байт
  nBits uint8  — оставшихся бит в cur

  readBit():
    если nBits == 0: загрузить следующий байт
    вернуть бит на позиции (nBits - 1), уменьшить nBits
```

### Timestamp: Delta-of-Delta (DoD)

Идея: вместо абсолютного timestamp кодируем разницу разниц.

```
delta[i]  = ts[i] - ts[i-1]
dod[i]    = delta[i] - delta[i-1]
```

Для метрик с постоянным интервалом (например, каждые 15 секунд) `dod == 0` почти
всегда — это 1 бит на точку вместо 64.

**Формат в битовом потоке:**

```
Первая точка:   ts[0]   — raw int64, 64 бита
Вторая точка:   delta1  — raw int64, 64 бита (ts[1] - ts[0])
Начиная с i=2:  DoD-кодирование по таблице:

  Диапазон dod          Префикс   Payload          Итого
  ──────────────────────────────────────────────────────
  dod == 0              0         —                1 бит
  -63 .. 64             10        dod+63, 7 бит    9 бит
  -255 .. 256           110       dod+255, 9 бит   12 бит
  -2047 .. 2048         1110      dod+2047, 12 бит 16 бит
  иначе                 1111      raw int64        68 бит
```

Payload кодирует `dod` как беззнаковое число со смещением (offset encoding),
чтобы избежать знакового расширения.

---

### Value: XOR-кодирование

Идея: XOR двух соседних `float64` имеет мало значащих бит, если значения близки.

```
xor = Float64bits(v[i]) ^ Float64bits(v[i-1])
```

Значащие биты XOR — это биты между первым и последним ненулевым битом.
Храним только их, описывая блок через `leading` и `trailing` нулевые биты.

**Формат:**

```
xor == 0:
  '0'                           — 1 бит (значение не изменилось)

xor != 0:
  '1'

  Новый блок ('1'):
    leading   — 5 бит (кол-во ведущих нулей, max 31)
    (meaningful-1) — 6 бит
    xor >> trailing — meaningful бит

  Переиспользовать блок ('0'):
    xor >> prevTrailing — prevMeaningful бит
```

Блок переиспользуется если `leading >= prevLeading AND trailing >= prevTrailing`,
то есть значащие биты нового XOR вписываются в ранее описанный диапазон.

**Состояние между точками (`xorState`):**

```go
type xorState struct {
    prevValBits  uint64  // предыдущее значение в битах
    prevLeading  uint8   // ведущие нули предыдущего блока
    prevTrailing uint8   // хвостовые нули предыдущего блока
    hasBlock     bool    // был ли хоть один ненулевой XOR
}
```

### Полный формат сжатых данных

```
ts[0]       int64    64 бита  — абсолютный timestamp первой точки
val[0]      float64  64 бита  — первое значение
delta1      int64    64 бита  — ts[1] - ts[0]
val[1]      XOR-encoded       — второе значение
[для i ≥ 2]
  DoD(ts[i])                  — 1..68 бит
  XOR(val[i])                 — 1..68 бит
```

## Формат chunk-файла v0x02

Header (24 байта) остался прежним, изменилось тело серии:

```
Версия 0x01 (raw):
  series_id      uint64
  labels_len     uint16
  labels         [labels_len]byte
  points_count   uint32
  points         [points_count × 16]byte   ← [ts int64][val float64]

Версия 0x02 (gorilla):
  series_id      uint64
  labels_len     uint16
  labels         [labels_len]byte
  points_count   uint32
  compressed_len uint32                    ← НОВОЕ
  gorilla_data   [compressed_len]byte      ← НОВОЕ
```

Reader поддерживает обе версии: старые chunk-файлы читаются без изменений.

## Известные ограничения

- **`leading` ограничен 31** (5 бит). Для XOR с >31 ведущими нулями лишние нули включаются
  в payload — декодирование корректно, незначительный overhead в редких случаях.
- **`n` при декодировании** должен точно совпадать с числом точек при кодировании.
  Это число хранится в `points_count` поля chunk-файла — reader передаёт его сам.
- **Случайные данные** практически не сжимаются (~1x): Gorilla ориентирован
  на реальные метрики с плавным изменением.
